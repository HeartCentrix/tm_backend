## Why Outlook doesn't show our contact (root cause hypothesis)

scanpst on agab.pst reports:
```
!!Computed cbAMapFree of 235328, but header has 474944
!!Hierarchy Table for 122, row doesn't match sub-object: irow = 0, RowID = 2223
!!Contents Table for 8002, row doesn't match sub-object: irow = 0, RowID = 200024
   Failed to add row to the FLT, irow = 0, RowID = 200024
```

"Failed to add row to the FLT" is the **visibility-blocking error**. FLT = Folder Lookup Table; Outlook refuses to register the contact.

The "row doesn't match sub-object" check compares Contents-TC row cells against the message PC. Our row sets CEB bits for 0x0057, 0x0058, 0x0E17, 0x0E08 (always-set per our 29-col schema), and the **REF PC doesn't have these tags at all** — so Outlook's contact-aware validator sees a row claiming "MessageStatus=0" that the PC contradicts by omission. Adding these to our PC (M12.1) didn't help because they're still PC-foreign tags from Outlook's perspective.

## Action plan to make contacts work (ordered by likelihood)

1. **Match the row to what Outlook expects for IPM.Contact** — stop setting CEB bits 2/10/13/14/19 (MessageStatus, MessageSize, MessageToMe, MessageCcMe, ChangeKey) on contact rows. This means:
   - Plumb an `emitOptionalFixedCells` flag into `ContentsTcRow` (or a `bool isContactRow`)
   - Have `buildFolderContentsTc` honour the flag and skip those CEB bits when false
   - **Simultaneously remove from the contact PC**: 0x0057, 0x0058, 0x0E17, 0x0E08, 0x0E1B, 0x1080, 0x3002, 0x3003, 0x3013
2. **MessageFlags 0x01 → 0x09** (mfRead | mfUnsent) — Outlook-canonical for locally-created contact items. *Already edited in contact.cpp; awaiting rebuild.*
3. **Replace literal-SMTP SearchKey with opaque hash** — change PC + row to use `deriveMessageSearchKey()` (FNV-1a-based, already used by mail/event) instead of `deriveSearchKey()` for contacts.
4. **Change InternetCpid 65001 → 20127** (US-ASCII) — Outlook canonical for contacts.
5. **Add `0x0E2F PR_LTP_PARENT_NID` = parent folder NID** to contact PC — gives Outlook the in-PC parent linkage it expects.
6. **Add minimal originator/creator fields**: 0x3FFA (creator name = displayName), 0x4038/0x4039 (simple display names), 0x3FFD (codepage 1252).
7. **Beef up the Contacts FOLDER PC**: add 0x3007/0x3008 (creation/mod time), 0x300B (folder SearchKey), 0x7B9E (container hierarchy name), 0x3900 (DisplayType).

Items 1-2 are the most likely to actually fix Outlook visibility. Items 3-7 are alignment work to also pass scanpst cleanly.
